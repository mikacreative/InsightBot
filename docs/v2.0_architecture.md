# InsightBot 多任务架构说明

> 更新日期：2026-04-16

## 背景

v1.x 为单一任务、单一频道架构：所有 RSS 源共享一个池，任务定义和频道配置内联在 `config.content.json` 中，调度依赖系统 cron。

这份文档记录的是当前正式版本所采用的多任务架构基线。  
历史上它曾在开发阶段被称为 `v2.0`，但该命名不再代表正式 release 编号。

架构目标：**多任务 × 多频道 × 内置调度 × 调试友好**。

## 核心设计决策

### 1. 任务独立化（Option A）

每个任务有自己独立的 feeds、pipeline_config、channels、schedule。任务之间互不干扰，可以有不同的推送时间、不同的频道组合。

**为什么不选共享池？** 共享池适合内容完全相同的场景，但营销简报的场景中，不同任务（如"每日早报"vs"每周深度周报"）通常需要不同的 RSS 源和筛选标准。

### 2. 调度器内置（Option A）

采用 Python 前台阻塞循环 (`scheduler.run_loop()`)，由 `python -m insightbot.cli` 作为常驻进程运行。

**为什么不选外部 cron？** 外部 cron 难以做幂等控制，且日志分散。"运行所有已启用任务"功能在 cron 模式下需要额外胶水代码。生产上只需守护一个 CLI 进程即可。

**幂等保护**：每次任务触发后 70 秒内不会重复运行，防止调度间隔过短或手动触发的重复。

### 3. Channels 独立配置（Option C）

企业微信凭证存在独立的 `channels.json`，与 `tasks.json` 解耦。

**好处**：
- 同一频道可被多个任务引用（DRY）
- 频道联通性测试可以独立于任务调试进行
- 敏感信息分离，config.content.json 可安全纳入 Git

## 数据流

```
CLI / Scheduler
    └─► task_runner.run_task(task_id, dry_run=False)
            ├─► config.load_tasks_config(task_id)   # 组装运行时配置
            ├─► editorial_pipeline.run_editorial_pipeline / smart_brief_runner.run_task
            │       └─► 返回 { ok, final_markdown, stage_results, ... }
            └─► send_to_channel(channel_id, content)   # 仅 dry_run=False 时
                    └─► channels.get_channel(channel_id).send(content)
                            └─► send_markdown_to_app(cid, secret, agent_id, content)
```

## 关键不变性

- `run_editorial_pipeline()` 和 `smart_brief_runner.run_task()` **不直接发送任何频道消息**，只返回 `final_markdown`
- 频道发送统一由 `task_runner` 管理，dry_run=True 时跳过所有发送
- `INSIGHTBOT_DRY_RUN` 环境变量可在任何时刻强制所有发送进入测试模式

## 控制台信息架构

控制台已从“全局单任务后台”调整为“任务中心后台”：

- 侧边栏提供当前任务切换，后续概览、Prompt Debug、RSS 健康度、信源发现、Dry Run 都默认工作在该任务上下文
- `任务管理` 页不只负责任务 CRUD，也负责维护该任务的 feeds、search、pipeline_config、channels、schedule
- `运行日志` 与 `No Push Diagnosis` 优先展示当前任务相关信息，减少多任务场景下的排查噪音

这意味着 `tasks.json` 不再只是调度器的后端数据结构，也成为控制台的主要产品模型。

## 迁移路径

首次启动当前版本时（检测到 `channels.json` / `tasks.json` 不存在）：

```
migrate_from_v1(bot_dir)
    ├─► 读取 config.content.json + config.secrets.json
    ├─► 生成 channels.json（wecom_main）
    └─► 生成 tasks.json（daily_brief，包含现有 feeds、search、pipeline_config）
```

存量用户无感知，无需手动迁移。
