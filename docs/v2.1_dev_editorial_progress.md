# InsightBot v2.1 进展说明（基于 dev-editorial）

> 状态：进行中  
> 基线分支：`origin/dev-editorial`  
> 当前整合分支：`codex/dev-editorial-state-adapt`

## 1. 这次整合的目标

`dev-editorial` 已经完成了 v2.0 的核心底座：

- 多任务模型：`tasks.json`
- 多频道模型：`channels.json`
- 统一执行入口：`task_runner.py`
- 内置调度：`scheduler.py`
- 任务型控制台：`scripts/app.py`
- `editorial pipeline` 生产化

本轮整合不是重做 v2.0，而是在这个底座上补 v2.1 的“状态层”和“运营闭环”。

本轮目标只有四个：

1. 让管理员明确知道任务能不能跑
2. 让管理员明确知道最近有没有真正发出去
3. 让配置变更和验证结果建立失效关系
4. 让“验证与诊断”页能直接衔接 Prompt Debug

---

## 2. 当前已完成的能力

### 2.1 任务级运行历史

新增：

- `insightbot/run_history.py`
- `insightbot/paths.py` 中的任务运行历史路径

能力：

- 每次 `task_runner.run_task()` 都会写结构化运行记录
- 同时记录正式运行和 `dry_run`
- 支持按 `task_id` 查询：
  - 最近一次运行
  - 最近一次成功发送
  - 最近若干次任务记录

记录字段包含：

- `task_id`
- `task_name`
- `started_at`
- `ended_at`
- `ok`
- `dry_run`
- `pipeline`
- `candidate_count`
- `selected_count`
- `channel_results`
- `error`

这解决了 v2.0 里“很多状态仍需靠日志猜”的问题。

### 2.2 任务可运行性校验

新增：

- `insightbot/task_validation.py`

能力：

- 校验任务板块、RSS、频道、schedule、search、pipeline 配置
- 返回统一结构：
  - `is_runnable`
  - `status`
  - `issues`
  - `summary`

当前主要错误类型：

- 没有板块
- 板块没有 RSS
- 没有频道
- 频道不存在
- 缺少 schedule
- search 开启但 query 为空
- pipeline 配置不完整

这让控制台第一次具备了“保存后就知道能不能跑”的基础能力。

### 2.3 任务状态与需重验机制

新增：

- `insightbot/task_state.py`
- `insightbot/task_health_store.py`

能力：

- 根据当前任务运行时配置生成 `config_revision`
- 保存任务级状态：
  - `config_revision`
  - `needs_revalidation`
  - `last_validated_revision`
- 当任务配置、频道、全局 system prompt、输出规则、版式等发生变更时：
  - 标记该任务或全部任务 `needs_revalidation = true`
  - 清理对应任务健康缓存

这解决了“用户改了配置，却还在看旧验证结果”的问题。

### 2.4 任务级健康缓存

新增：

- `data/task_health/<task_id>.json`

能力：

- RSS 健康快照不再只存在于当前 Streamlit session
- 切换任务后，健康状态可持续读取
- 刷新健康度后会写回任务级缓存
- 验证成功后会同步清理 `needs_revalidation`

### 2.5 概览页状态化

控制台概览页已补齐以下信息：

- 任务状态
- 可运行性
- 最近一次运行
- 最后成功发送
- 最近调试动态
- 异常摘要

这意味着首页已经从“几个指标”开始向“任务运营台首页”转变。

### 2.6 验证与诊断页合并

控制台里原本分散的：

- RSS 健康度
- No Push Diagnosis
- 最近相关日志

已经合并到 `🧪 验证与诊断` 页。

当前该页已具备：

- 最近运行 / 最近成功发送 / 最近健康检查 / 最近调试摘要
- 任务级 `needs_revalidation` 提示
- RSS 健康度总览
- No Push Diagnosis 卡片
- 当前任务相关日志摘要

### 2.7 验证页与 Prompt Debug 闭环

这是当前分支相较原始 `dev-editorial` 最明显的 UX 提升之一。

现在诊断卡片已经支持：

- 直接聚焦问题板块
- 将板块预设到 Prompt Debug
- 直接抓候选内容到调试台
- 在验证页中看到该板块最近一次 Prompt Debug 摘要

当前链路已经可以做到：

1. 看到“为什么今天没推送”
2. 聚焦具体板块
3. 准备候选
4. 跳去 Prompt Debug 继续排查

这比 v2.0 原始状态更接近“运营闭环”。

## 5. 现阶段判断：是否可并到 dev-editorial

结论：**可并。**

原因：

1. 本轮改动顺着 v2.0 的任务模型长出来，没有引入平行配置体系
2. 状态层主要挂在 `task_id`、`task_runner`、任务型控制台之上，职责边界清晰
3. 对生产流程的影响有限，主要是新增状态记录、验证提示和控制台闭环
4. 回归测试结果稳定

当前仍然存在但不阻塞合并的问题：

- `scripts/app.py` 继续膨胀，后续应拆到 `scripts/ui/*.py`
- 运行历史目前采用 `jsonl`，未来如查询需求增多可迁到 sqlite
- 创建任务后的空状态向导尚未补上
- 验证页虽然已形成闭环，但还未把 Dry Run 进一步嵌入同一操作面板

---

## 6. 合并后推荐的下一步

建议下一阶段优先级：

1. 拆分 `scripts/app.py`
2. 补创建任务空状态向导
3. 把 Dry Run 更深地并入“验证与诊断”页
4. 视任务量决定是否把运行历史迁为 sqlite

---

## 7. 关于这次强制覆盖

如果将当前整合分支强制推送到 `origin/dev-editorial`，意味着：

- 远端 `dev-editorial` 会直接前进到当前整合分支的提交
- 后续继续开发应以新的 `origin/dev-editorial` 为基线
- 旧的远端 `dev-editorial` 提交历史不会出现在分支顶端，但仍可通过提交哈希找回

这次强制覆盖的目的不是回退 v2.0，而是把“v2.0 底座 + v2.1 状态层适配”直接立成新的共享基线。
