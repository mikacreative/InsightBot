# InsightBot 管理台下一版本 PRD

> **当前阶段版本：** v0.2.0  
> **下一阶段目标版本：** v0.3.0  
> **文档状态：** Draft  
> **制定日期：** 2026-04-07

---

## 1. 当前阶段定义

### 1.1 v0.2.0 已完成能力

当前阶段版本定义为 **v0.2.0**，它代表产品已经从“可运行的 RSS + AI 推送工具”进入“可由管理员安全调优的控制台阶段”。

`v0.2.0` 的核心能力包括：

- 双层配置结构：`config.content.json` + `config.secrets.json`
- 控制台不再直接编辑 AI 运行时敏感参数
- Prompt Debug Console：
  - 单板块抓取候选内容
  - 草稿 Prompt 试跑
  - 空结果 / 失败 / 成功状态展示
  - 当前版 vs 草稿版并排对比
- 工程测试收缩并对齐到当前 JSON-based runner

### 1.2 当前阶段仍然存在的问题

- 管理员还无法在控制台内直接看到 RSS 源健康度，只能通过脚本或日志判断
- Prompt 调试虽然可用，但没有“留痕”和“最近调试记录”
- 管理台信息结构仍然偏“配置编辑器”，尚未完全进化成“运营控制台”
- 管理员很难快速判断“为什么今天无推送”是源没更新、源挂了、还是 Prompt 过严

---

## 2. v0.3.0 目标

`v0.3.0` 的目标不是继续扩功能面，而是把管理台做成真正可用于日常运营的“管理员控制台”。

这一版本重点解决三件事：

1. **RSS 源健康度可视化**
2. **管理台进一步优化为运营工作台**
3. **调优留痕与最近调试记录**

一句话定义：

> 让管理员在一个界面里看清“源是否健康、Prompt 是否有效、最近做过什么调优”。

---

## 3. 核心用户与使用场景

### 3.1 核心用户

- 内部管理员
- 运营负责人
- 负责内容质量和情报输出稳定性的维护者

### 3.2 关键使用场景

#### 场景 A：今天为什么没推送？

管理员进入管理台后，可以快速判断：

- RSS 源最近 24h 是否有更新
- 哪些源不可达
- 哪些板块候选为 0
- 哪些板块候选很多但 Prompt 全拦截

#### 场景 B：我想调一个 Prompt，但不想破坏线上输出

管理员可以：

- 看当前 Prompt
- 写草稿
- 用同一批候选内容进行对比试跑
- 保存前查看最近几次调试记录

#### 场景 C：我想判断某个源要不要继续保留

管理员可以看到：

- 最近抓取状态
- 最近更新时间
- 近 24h / 近 7d 的有效文章数
- 是否经常超时、报错、长期无更新

## 4. v0.3.0 核心功能设计

### 4.1 新能力一：RSS 源健康度面板

这是 `v0.3.0` 的第一优先级功能。

#### 目标

把当前脚本化的 RSS 健康度检查能力搬进管理台，让管理员无需 SSH 登录服务器也能看到源状态。

#### 展示层级

##### 板块层

每个板块展示：

- RSS 源总数
- 正常源数量
- 近 24h 有更新的源数量
- 报错源数量
- 长期无更新源数量

##### 单源层

每个源展示：

- 源名称 / URL
- 当前状态：正常 / 无更新 / 错误
- 最近一次检查时间
- 最近一篇文章发布时间
- 近 24h 有效文章数
- 总抓取文章数
- 错误原因（如有）

#### 状态定义

- **正常**：可达，且近 24h 有更新
- **无更新**：可达，但近 24h 无更新
- **错误**：不可达、解析失败、超时或结构异常

#### 交互

- 手动刷新全量健康度
- 按板块筛选
- 仅查看异常源
- 仅查看“7 天无更新”源

#### 数据来源

优先复用现有 `debug_rss_check.py` 的检查逻辑，抽成共享 helper，避免控制台和脚本各维护一套规则。

---

### 4.2 新能力二：管理台进一步优化

目标是让控制台从“配置编辑器”进化成“运营驾驶舱”。

#### 优化方向 A：统一的状态摘要区

在管理台首页或顶部增加一组摘要卡片：

- 今日板块数
- 异常 RSS 源数
- 最近一次任务运行时间
- 最近一次运行结果：成功 / 空 / 部分异常
- 今日候选总条数

#### 优化方向 B：更清晰的工作流入口

管理台中形成三个明确入口：

1. 内容配置
2. Prompt 调试
3. 源健康度

避免管理员在同一页面里混淆：

- 这是“改配置”
- 还是“查健康”
- 还是“调 Prompt”

#### 优化方向 C：无推送诊断入口

增加一个“为什么今天没推送？”快捷区，优先展示：

- 哪些板块没有候选
- 哪些板块有候选但全被拦截
- 哪些 RSS 源报错

这是管理台最重要的运营诊断入口之一。

---

### 4.3 新能力三：调优留痕

当前不做复杂的发布流，但需要最低限度的“调试留痕”。

#### 目标

让管理员知道：

- 最近调过哪些板块
- 用的是什么草稿
- 调试结果是更好还是更差

#### 最小实现范围

记录最近 N 次调试结果，建议先做最近 20 条。

每条记录包含：

- 时间
- 板块
- 使用的是“当前版试跑”还是“草稿版对比”
- 候选条数
- 命中条数
- 结果状态：成功 / 空 / 错误
- 草稿 Prompt 摘要（截断展示）

#### 展示位置

放在 Prompt Debug Console 下方或单独的“最近调试记录”区域。

#### 边界

本阶段只做“最近记录”，不做：

- 完整版本历史
- 回滚机制
- 审批流
- 多人协作评论

## 5. 交互与信息架构建议

### 5.1 管理台建议结构

建议在 `v0.3.0` 把管理台逐步整理为以下结构：

- **概览**
  - 系统状态
  - 最近运行结果
  - 异常摘要
- **板块与信源管理**
  - 编辑 feeds / prompt
- **Prompt 调试台**
  - 当前版 / 草稿版对比
  - 最近调试记录
- **RSS 健康度**
  - 板块级健康度
  - 单源级状态
- **运行日志**
  - 深度日志与原始任务轨迹

### 5.2 为什么 RSS 健康度应该进管理台

答案是：**应该，而且优先级很高。**

原因：

- 它直接解释“无推送”的大部分问题
- 它比纯日志更适合运营同学理解
- 它能帮助管理员清理长期失效源
- 它能减少对服务器命令行和人工排查的依赖

所以在产品层面，RSS 健康度不是“可选增强”，而是管理台成为运营工具的必要组成部分。

---

## 6. 非目标

`v0.3.0` 明确不做：

- Prompt 发布审批流
- Prompt 版本库
- 自动回滚
- 多环境（dev/staging/prod）切换
- 长周期运营分析 dashboard

这类能力可以放到后续 `v0.4.x` 再讨论。

---

## 7. 验收标准

### 7.1 RSS 健康度

- 管理台内可查看所有 RSS 源状态
- 可区分正常 / 无更新 / 错误
- 可快速定位异常板块与异常源

### 7.2 Prompt 调试留痕

- 最近调试记录可查看
- 可以知道一次调试是成功、空结果还是报错
- 可以知道草稿相对当前版是否更优

### 7.3 管理台可用性

- 管理员不登录服务器，也能完成大多数日常排查
- “今天为什么没推送”能够在管理台内得到初步诊断
- 控制台信息结构比当前阶段更清晰，不再只是配置表单堆叠

---

## 8. 建议的开发优先级

1. **RSS 健康度共享逻辑抽取 + 控制台可视化**
2. **Prompt 调试记录留痕**
3. **概览页 / 诊断摘要区**
4. **管理台信息架构整理**

---

## 9. 结论

`v0.2.0` 已经解决了“怎么安全调 Prompt”的问题。  
`v0.3.0` 要解决的是“怎么把日常运营看清楚、判断准、追得回”。

因此，下一阶段最正确的方向不是继续堆更多 Prompt 能力，而是让管理台真正具备：

- 看源健康
- 看调优结果
- 看最近发生过什么

这会让 InsightBot 从“一个能调的工具”更进一步变成“一个能稳定运营的系统”。
