# InsightBot 管理台下一版本 PRD

> **当前阶段版本：** v0.2.0  
> **下一阶段目标版本：** v0.3.0  
> **文档状态：** Draft  
> **制定日期：** 2026-04-07

---

## 1. 当前阶段定义

### 1.1 v0.2.0 已完成能力

当前阶段版本定义为 **v0.2.0**，它代表产品已经从“可运行的 RSS + AI 推送工具”进入“可由管理员安全调优的控制台阶段”。

`v0.2.0` 的核心能力包括：

- 双层配置结构：`config.content.json` + `config.secrets.json`
- 控制台不再直接编辑 AI 运行时敏感参数
- Prompt Debug Console：
  - 单板块抓取候选内容
  - 草稿 Prompt 试跑
  - 空结果 / 失败 / 成功状态展示
  - 当前版 vs 草稿版并排对比
- 工程测试收缩并对齐到当前 JSON-based runner

### 1.2 当前阶段仍然存在的问题

- 管理员还无法在控制台内直接看到 RSS 源健康度，只能通过脚本或日志判断
- Prompt 调试虽然可用，但没有“留痕”和“最近调试记录”
- 管理台信息结构仍然偏“配置编辑器”，尚未完全进化成“运营控制台”
- 管理员很难快速判断“为什么今天无推送”是源没更新、源挂了、还是 Prompt 过严

---

## 2. v0.3.0 目标

`v0.3.0` 的目标不是继续扩功能面，而是把管理台做成真正可用于日常运营的“管理员控制台”。

这一版本重点解决三件事：

1. **RSS 源健康度可视化**
2. **管理台进一步优化为运营工作台**
3. **调优留痕与最近调试记录**

一句话定义：

> 让管理员在一个界面里看清“源是否健康、Prompt 是否有效、最近做过什么调优”。

---

## 3. 核心用户与使用场景

### 3.1 核心用户

- 内部管理员
- 运营负责人
- 负责内容质量和情报输出稳定性的维护者

### 3.2 关键使用场景

#### 场景 A：今天为什么没推送？

管理员进入管理台后，可以快速判断：

- RSS 源最近 24h 是否有更新
- 哪些源不可达
- 哪些板块候选为 0
- 哪些板块候选很多但 Prompt 全拦截

#### 场景 B：我想调一个 Prompt，但不想破坏线上输出

管理员可以：

- 看当前 Prompt
- 写草稿
- 用同一批候选内容进行对比试跑
- 保存前查看最近几次调试记录

#### 场景 C：我想判断某个源要不要继续保留

管理员可以看到：

- 最近抓取状态
- 最近更新时间
- 近 24h / 近 7d 的有效文章数
- 是否经常超时、报错、长期无更新

## 4. v0.3.0 核心功能设计

### 4.1 新能力一：RSS 源健康度面板

这是 `v0.3.0` 的第一优先级功能。

#### 目标

把当前脚本化的 RSS 健康度检查能力搬进管理台，让管理员无需 SSH 登录服务器也能看到源状态。

#### 展示层级

##### 板块层

每个板块展示：

- RSS 源总数
- 正常源数量
- 近 24h 有更新的源数量
- 报错源数量
- 长期无更新源数量

##### 单源层

每个源展示：

- 源名称 / URL
- 当前状态：正常 / 无更新 / 错误
- 最近一次检查时间
- 最近一篇文章发布时间
- 近 24h 有效文章数
- 总抓取文章数
- 错误原因（如有）

#### 状态定义

- **正常**：可达，且近 24h 有更新
- **无更新**：可达，但近 24h 无更新
- **错误**：不可达、解析失败、超时或结构异常

#### 错误类型枚举

为了避免把所有异常都折叠成同一种“错误”，`v0.3.0` 需要在数据层区分以下错误类型：

- `timeout`：请求超时
- `unreachable`：网络不可达、连接拒绝、DNS 失败
- `parse_error`：内容返回成功，但 RSS/Atom 解析失败
- `invalid_feed`：返回内容不是合法 Feed，或缺失关键字段
- `unknown_error`：未归类异常

界面展示要求：

- 列表层仍然只显示“正常 / 无更新 / 错误”三态，降低信息噪音
- 详情层必须展示 `error_type` 和 `error_message`
- 管理员看到错误后，应能知道这是“等一会再试”还是“这个源需要处理/移除”

#### 交互

- 手动刷新全量健康度
- 按板块筛选
- 仅查看异常源
- 仅查看“7 天无更新”源

#### 检查频率与缓存策略

健康度检查默认不是页面加载时全量实时执行，而采用“缓存优先 + 手动刷新”的模式。

默认策略：

- 页面打开时，优先读取最近一次健康度缓存
- 缓存有效期：**5 分钟**
- 若缓存过期，页面先展示旧缓存并提示“数据可能已过期”
- 管理员点击“立即刷新”后，触发一次全量检查并覆盖缓存

为什么这样设计：

- 全量 RSS 检查耗时不稳定，页面初始化时强制实时检查会拖慢控制台体验
- 5 分钟缓存足够支持日常运营判断，又不会让状态陈旧过久
- 手动刷新可以让管理员在排障时主动拿到最新结果

异步刷新策略：

- 第一个版本先采用同步执行 + 进度提示，不引入后台任务系统
- 刷新期间界面显示“检查中”状态，避免管理员误以为页面卡死
- 后续如检查成本继续升高，再演进为后台异步任务

#### 数据来源

优先复用现有 `debug_rss_check.py` 的检查逻辑，抽成共享 helper，避免控制台和脚本各维护一套规则。

#### 技术拆分方案

这一部分需要先做一次小重构，再接 UI：

建议拆分为三层：

1. **源检查核心函数**
   - 输入：单个 RSS URL
   - 输出：标准化健康度结果对象
   - 从当前 `debug_rss_check.py` 中抽出 `check_feed()` 能力，沉到共享模块

2. **批量健康度聚合函数**
   - 输入：整个 `feeds` 配置
   - 输出：板块级汇总 + 单源级明细
   - 负责统计正常数、异常数、无更新数、错误类型分布

3. **缓存读写层**
   - 输入：最新健康度结果
   - 输出：JSON 缓存文件
   - 供脚本和控制台共用

建议新增一个共享模块，例如：

- `insightbot/feed_health.py`

`debug_rss_check.py` 改成调用这个共享模块，而不是继续独立维护自己的逻辑。

这样可以保证：

- CLI 检查脚本和控制台显示结果一致
- 后续如果要加定时健康检查，只需要复用同一套核心函数
- UI 改动不会反向污染脚本实现

### 4.2 新能力二：管理台进一步优化

目标是让控制台从“配置编辑器”进化成“运营驾驶舱”。

#### 优化方向 A：统一的状态摘要区

在管理台首页或顶部增加一组摘要卡片：

- 今日板块数
- 异常 RSS 源数
- 最近一次任务运行时间
- 最近一次运行结果：成功 / 空 / 部分异常
- 今日候选总条数

这些摘要数据的来源采用“缓存优先”：

- RSS 健康度摘要：来自最近一次健康度缓存
- 最近运行结果：来自 bot 日志的最近一次任务摘要
- 今日候选总条数：来自最近一次任务执行日志或当日 debug 检查结果，不要求精确到秒级实时

换句话说，概览页不是实时计算中心，而是“最近状态总览”。

#### 优化方向 B：更清晰的工作流入口

管理台中形成三个明确入口：

1. 内容配置
2. Prompt 调试
3. 源健康度

避免管理员在同一页面里混淆：

- 这是“改配置”
- 还是“查健康”
- 还是“调 Prompt”

#### 优化方向 C：无推送诊断入口

增加一个“为什么今天没推送？”快捷区，优先展示：

- 哪些板块没有候选
- 哪些板块有候选但全被拦截
- 哪些 RSS 源报错

这是管理台最重要的运营诊断入口之一。

#### 无推送诊断的交互流程

当管理员点击“为什么今天没推送？”后，界面按以下优先级展示诊断卡：

1. **源异常优先**
   - 如存在不可达、超时、解析失败的 RSS 源，优先展示
   - 下一步操作：
     - 跳转到 RSS 健康度面板
     - 快速定位异常源

2. **候选为 0**
   - 板块源可达，但近 24h 无有效候选
   - 下一步操作：
     - 跳转到板块源配置
     - 检查是否需要更换或补充源

3. **候选有值但 Prompt 全拦截**
   - 这是 Prompt 过严或源与板块目标不匹配的信号
   - 下一步操作：
     - 一键跳转到 Prompt Debug Console
     - 直接打开该板块做“当前版 vs 草稿版”对比

4. **任务运行异常**
   - 如日志中存在调用错误、AI 请求异常、推送异常
   - 下一步操作：
     - 跳转到运行日志

这个入口的目标不是展示所有日志，而是给管理员一个可执行的下一步。

### 4.3 新能力三：调优留痕

当前不做复杂的发布流，但需要最低限度的“调试留痕”。

#### 目标

让管理员知道：

- 最近调过哪些板块
- 用的是什么草稿
- 调试结果是更好还是更差

#### 最小实现范围

记录最近 N 次调试结果，建议先做最近 20 条。

#### 存储方案

`v0.3.0` 阶段建议采用 **JSON 文件持久化**，不引入 SQLite。

原因：

- 当前产品仍以单机部署为主，JSON 文件足够满足“最近 20 条”记录需求
- 实现成本低，便于部署和备份
- 未来如需要更多分析能力，再演进到 SQLite

建议路径：

- `data/prompt_debug_history.json`

写入责任：

- Prompt Debug Console 在每次“运行草稿 Prompt”或“对比当前 vs 草稿”完成后写入
- 只保留最近 20 条，超出后丢弃最旧记录

#### 数据模型

每条调试记录建议包含：

- `id`
- `created_at`
- `category`
- `mode`：`draft_run` / `compare`
- `candidate_count`
- `saved_selected_count`
- `draft_selected_count`
- `draft_status`
- `saved_status`
- `using_fallback_candidates`
- `draft_prompt_excerpt`

其中：

- `draft_run` 模式下，`saved_*` 字段可以为空
- `compare` 模式下，需同时记录当前版和草稿版结果

#### 为什么先保留 20 条

选择 `N = 20` 的原因：

- 足够覆盖管理员最近一段时间的调优上下文
- 不会让界面被历史记录淹没
- JSON 文件体积可控，不需要引入分页和复杂查询

后续若使用频率明显上升，再扩展为 50 条或切换到 SQLite。

每条记录包含：

- 时间
- 板块
- 使用的是“当前版试跑”还是“草稿版对比”
- 候选条数
- 命中条数
- 结果状态：成功 / 空 / 错误
- 草稿 Prompt 摘要（截断展示）

#### 展示位置

放在 Prompt Debug Console 下方或单独的“最近调试记录”区域。

#### 异常告警（延后但预留）

`v0.3.0` 不做主动通知型告警，但需要在 PRD 中预留这个方向：

- 若某个 RSS 源连续 N 次失败
- 若某个关键板块连续多日候选为 0
- 若多板块连续多日被 Prompt 全拦截

后续可以考虑：

- 控制台高亮提醒
- 企业微信管理员通知
- 日报摘要中附带系统健康度提示

本阶段先不做外部通知，只做界面内可见的异常提示。

#### 边界

本阶段只做“最近记录”，不做：

- 完整版本历史
- 回滚机制
- 审批流
- 多人协作评论

## 5. 交互与信息架构建议

### 5.1 管理台建议结构

建议在 `v0.3.0` 把管理台逐步整理为以下结构：

- **概览**
  - 系统状态
  - 最近运行结果
  - 异常摘要
- **板块与信源管理**
  - 编辑 feeds / prompt
- **Prompt 调试台**
  - 当前版 / 草稿版对比
  - 最近调试记录
- **RSS 健康度**
  - 板块级健康度
  - 单源级状态
- **运行日志**
  - 深度日志与原始任务轨迹

### 5.2 为什么 RSS 健康度应该进管理台

答案是：**应该，而且优先级很高。**

原因：

- 它直接解释“无推送”的大部分问题
- 它比纯日志更适合运营同学理解
- 它能帮助管理员清理长期失效源
- 它能减少对服务器命令行和人工排查的依赖

所以在产品层面，RSS 健康度不是“可选增强”，而是管理台成为运营工具的必要组成部分。

## 6. 非目标

`v0.3.0` 明确不做：

- Prompt 发布审批流
- Prompt 版本库
- 自动回滚
- 多环境（dev/staging/prod）切换
- 长周期运营分析 dashboard

这类能力可以放到后续 `v0.4.x` 再讨论。

---

## 7. 验收标准

### 7.1 RSS 健康度

- 管理台内可查看所有 RSS 源状态
- 可区分正常 / 无更新 / 错误
- 可快速定位异常板块与异常源

### 7.2 Prompt 调试留痕

- 最近调试记录可查看
- 可以知道一次调试是成功、空结果还是报错
- 可以知道草稿相对当前版是否更优

### 7.3 管理台可用性

- 管理员不登录服务器，也能完成大多数日常排查
- “今天为什么没推送”能够在管理台内得到初步诊断
- 控制台信息结构比当前阶段更清晰，不再只是配置表单堆叠

---

## 8. 建议的开发优先级

1. **RSS 健康度共享逻辑抽取 + 控制台可视化**
2. **Prompt 调试记录留痕**
3. **概览页 / 诊断摘要区**
4. **管理台信息架构整理**

---

## 9. 结论

`v0.2.0` 已经解决了“怎么安全调 Prompt”的问题。  
`v0.3.0` 要解决的是“怎么把日常运营看清楚、判断准、追得回”。

因此，下一阶段最正确的方向不是继续堆更多 Prompt 能力，而是让管理台真正具备：

- 看源健康
- 看调优结果
- 看最近发生过什么

这会让 InsightBot 从“一个能调的工具”更进一步变成“一个能稳定运营的系统”。
